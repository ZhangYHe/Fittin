## Context

当前训练记录页的主要问题不是视觉风格，而是信息结构和交互优先级。页面把顶部标题、动作英雄卡、横向动作切换、当前组控制、组进度、完成训练按钮垂直堆叠，导致在移动端首屏内无法完整容纳。用户补充的设计约束很明确：
- 常见移动端下不需要上下滑动
- 顶部动作预览明显过大，需要压缩
- 动作切换整合进动作预览，而不是单独占一行
- 不要“方案”字段
- 不要“已完成几组 / 上次几kg乘几 / 当前聚焦动作与当前组控制台”
- 标题合并成 `W1D1-深蹲与拉` 这种训练上下文形式
- 不需要“重置次数”功能
- 现有按键逻辑存在错误或缺失，需要修正

## Goals / Non-Goals

**Goals:**
- 将训练记录页压缩为单屏可用的移动端控制台
- 简化顶部信息，只保留最必要训练上下文、动作名、层级、目标
- 将动作切换合并为动作预览中的菜单动作
- 修正主要交互：加次数、减次数、记录组、组推进、动作切换
- 保持 premium minimal 风格，但让紧凑优先于炫技

**Non-Goals:**
- 不重写整个训练状态机
- 不新增休息计时器或复杂历史面板
- 不扩展到分析页、设置页或计划管理页
- 不引入多步骤训练记录流程

## Decisions

### 1. 页面采用“单卡主控台 + 底部进度与完成按钮”的固定结构

训练记录页改为 4 段：
1. 顶部上下文栏：`WnDn-训练名` + 关闭按钮
2. 紧凑动作预览卡：动作名、当前第几组/共几组、层级、目标、动作切换入口
3. 当前组主控台：减次数、中央主按钮 `+1`、记录组按钮、重量调整
4. 底部紧凑进度条 + 完成本次训练按钮

这样可以把训练页压缩为真正的一屏控制台，滚动只作为极端兜底而不是常态。

### 2. 顶部动作预览卡只保留当前决策所需信息

动作预览卡移除：
- 方案
- 已完成几组
- 上次重量 x 次数
- 冗余副标题

只保留：
- 当前动作名称
- `第 N 组 / 共 M 组`
- 层级
- 目标（重量 + 次数 / AMRAP）
- 动作切换图标

这能显著降低垂直高度，并避免用户在记录时读取无关元数据。

### 3. 动作切换通过 anchored menu / dropdown sheet 实现

动作切换不再单独占据“动作切换”区域。动作预览卡右上角提供小图标按钮，点击后弹出带轻微缩放和淡入的 anchored menu。

菜单内容：
- 当前 workout 的全部动作
- 当前动作高亮
- 每个动作显示 tier + 简名 + 完成进度

如果空间不足，可退化为底部 modal sheet，但默认优先 anchored menu，因为它更紧凑，也更符合“高级感”要求。

### 4. 当前组主控台改成明确的动作型按钮语义

当前组控制区保留：
- `-1 reps`
- 中央 `+1` 主按钮
- `Log Set` / `完成本组`
- 重量增减

移除：
- `reset reps`

行为约束：
- `-1` 只减少当前组已填次数，不能小于 0
- `+1` 增加当前组次数
- `Log Set` 立即把当前组标记完成，并推进到下一组；若已是最后一组则只完成当前动作，不自动结束整次训练
- 重量调整只影响当前组实际记录重量，不篡改目标重量

### 5. 组进度压缩为线性节点，不展开 dense set list

保留简化进度条，显示每组圆点和编号，当前组高亮，完成组实心。它只承担“当前在哪一组”的作用，不承载编辑面板。

### 6. 上下文标题统一为程序化格式

顶部标题统一显示为：
- `W1D1-深蹲与拉`
- `W3D2-Bench & Pull`

规则：
- `Wn` 从训练状态/计划周次推导
- `Dn` 从 workout/day label 推导
- 后半段为本次训练名

如果某计划无法准确推导周次，则回退到 `D1-深蹲与拉`，但默认设计以 `WnDn` 为标准。

## Risks / Trade-offs

- 单屏约束会压缩一些次要信息，需要接受“更少说明、更多动作”的取舍。
- anchored menu 若动画处理不好，会显得像普通弹窗而非 premium 交互。
- 不展示上次记录会减少一部分参考信息，但能换来更清晰的主流程；如果后续需要，可以作为次级展开信息补回。
